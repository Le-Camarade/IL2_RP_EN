# IL-2 Career Narrator — A PWCG Narrative Roleplay Module

*An experiment in turning raw campaign data into living story.*

---

## What Is This?

This is a small personal project that connects **IL-2 Sturmovik Great Battles** campaign data — generated by [PWCG (Pat Wilson's Campaign Generator)](https://www.pwcampaignmanager.com/) — to a large language model AI (LLM). The tool used here is **Claude** by Anthropic, via a CLI called **Claude Code** — but the concept is not tied to any specific model. Any LLM that can be driven from a terminal with read/write access to your local files would work: GPT-4 via the OpenAI CLI, a local model run through [Ollama](https://ollama.com/), Gemini via the Google CLI, and so on. The only hard requirement is terminal access — the AI needs to be able to read your PWCG and IL-2 files directly, and write the output back to disk.

The goal is simple: instead of flying a campaign and reading numbers, you fly a campaign and *live a story*. The AI reads your mission data, asks you what happened, and turns it all into:

- A proper **pre-mission briefing**, delivered in character by an Intelligence Officer
- An **interactive debriefing** after the mission — the IO interrogates you, then cross-references your account with the actual flight logs
- A **RAF-style Combat Report** filed automatically after debrief
- A **dispersal hut** scene where your squadron mates talk, argue, drink, and react to events
- **Pilot dossiers** that evolve with every mission — wounds, kills, personality
- A **squadron journal** written in the style of Pierre Clostermann's *The Big Show*

Everything is stored as plain Markdown files on your hard drive. No cloud, no subscription beyond whatever AI access you use.

---

## Important Disclaimer Upfront

**This is an experiment, not a finished product.**

This module is functional in its current form, but it requires you to be comfortable working in a terminal, has no graphical interface whatsoever, and still needs significant development before it could be considered convenient for general use. Think of it as a proof of concept: *this kind of thing is possible, it produces results worth reading, and someone more determined than me could build something much more polished on top of it.*

I'm sharing it because the output surprised me — the journal entries, the combat reports, the IO character — and I thought some of you might find the concept interesting, even if the tooling is rough.

## How Does It Actually Work?

This is the part that requires some explanation, because it is not magic — it is a very specific, carefully structured set of instructions given to an AI. Understanding the pipeline helps you understand both what it can do and where it falls apart.

### Step 1 — The Data Sources

PWCG generates rich data files when it creates a campaign and each mission. This module reads three types of data:

**1. PWCG JSON files** — When PWCG creates your campaign, it stores everything: your squadron's pilots (name, rank, skill level, victory count, status), the mission parameters (type, objective, altitude, weather, enemy composition), and post-mission reports. These are plain JSON files on your hard drive.

**2. The .Mission file** — PWCG generates a mission file for IL-2 that contains the full structure: waypoints, objectives, aircraft, weather, everything. This is used for the briefing.

**3. IL-2 Mission Report logs** — When you have `mission_text_log = 1` enabled in IL-2's `startup.cfg`, the game generates text log files after every flight. These logs record every event in the mission with timestamps: when aircraft spawned, who hit who, who was damaged, who was killed, who took off, who landed. This is the ground truth that the AI uses to verify your debrief account.

### Step 2 — The Scripts and the Raw Logs

Two small Python scripts parse this raw data and convert it into structured JSON that the AI can easily process:

- `parse_mission_report.py` — reads the IL-2 logs and reconstructs the mission timeline
- `parse_pwcg.py` — reads the PWCG JSON files and extracts the relevant data

These scripts run silently at the start of the debrief workflow. However, **the scripts are a supplement, not the primary source**. In practice, before running them, the AI reads a selection of the raw `missionReport[N].txt` files directly — specifically to extract aircraft IDs, identify kills, and track allied casualties. This is because the scripts can miss combat events when enemy aircraft are defined as groups (AType:11 entries), which is common in PWCG-generated missions. The raw logs are always authoritative; the scripts provide a useful starting summary.

A `post_mission.py` utility handles two additional tasks:
- `fix-lang` — copies the mission's `.eng` description file to all other language variants (`.fra`, `.ger`, `.rus`, etc.) that PWCG expects. Run at the start of every briefing.
- `journal YYYYMMDD "text"` — injects a journal entry into PWCG's `CampaignLog.json`, keeping the in-game campaign log in sync with the narrative files.

### Step 3 — The AI and the Prompts

Here is where the concept requires a moment of explanation for anyone new to working with AI language models.

A **large language model** (LLM) like Claude does not "remember" anything between conversations by default. Each time you open a new session, it starts from scratch. What makes it useful for a project like this is **context**: if you give it the right data, the right instructions, and the right persona at the start of the conversation, it will behave consistently and produce coherent output.

This module works by giving Claude three things at the start of each workflow:

1. **A persona** — "You are the Intelligence Officer of an RAF squadron, 1944-45. You do this every day. You are professional, dry, and precise." This is defined in files called *system prompts* stored in `resources/`.

2. **The data** — The parsed mission data, the pilot dossiers, the squadron journal, the campaign date. All fed in at the start.

3. **A structure** — The debriefing, for example, follows six specific phases. The IO asks about the initial situation first, then the engagement, then casualties, then confirmation of claims. This prevents the AI from jumping to the end and producing generic output.

The result is an AI that *acts* like a consistent character, *knows* what actually happened in your mission (from the logs), and *writes* in a specific register that matches the period.

### Step 4 — Persistent Storage

After each workflow, the AI writes files to disk:
- A combat report in `missions/combat_reports/`
- Updated pilot dossiers in `personnel/`
- A journal entry in `squadron/journal.md`
- Updated squadron statistics in `squadron/tableau-de-bord.md`

On the next session, these files are re-read, so the AI has continuity. It knows Dusaussoy is wounded. It knows Vitry was corrected and is learning. It knows what was said in the dispersal hut last night.

This is **not** automatic memory built into the AI. It is a manual persistence system built on top of an AI that otherwise forgets everything. Files in, files out — that is the architecture.

## What Do You Need to Use It?

### Prerequisites

| Requirement | Details |
|-------------|---------|
| **IL-2 Sturmovik Great Battles** | Any version with PWCG support |
| **PWCG** | Installed and configured for your IL-2 version |
| **`mission_text_log = 1`** | Set in `startup.cfg` under `[KEY = system]` — without this, there are no IL-2 logs to parse |
| **Python 3.x** | For the two parsing scripts |
| **A terminal-driven LLM** | Claude Code (used here), or any equivalent — see below |
| **A terminal** | Everything runs in a command-line interface |

### Which AI Tool?

The critical requirement is not *which* AI you use, but *how* you access it: the model must be able to read files from your hard drive, run scripts, and write output back to disk — all from a terminal. A chat interface (a browser tab, a phone app) will not work, because it cannot reach your PWCG or IL-2 data.

Options that should work in principle:

- **Claude Code** (used here) — CLI by Anthropic, available at [claude.ai/code](https://claude.ai/code). Requires a Claude Pro subscription or an Anthropic API key. For reference, a typical debrief session is not expensive on pay-as-you-go pricing.
- **OpenAI CLI / Codex CLI** — similar concept, terminal-based, accesses local files
- **Gemini CLI** — Google's equivalent, recently released
- **Local models via Ollama** — runs entirely offline, no subscription, but output quality will depend heavily on the model you use and your hardware

The `CLAUDE.md` file at the root of this project is the instruction file Claude reads. If you use a different tool, you would rename or adapt it to whatever that tool expects as its project instructions file. The structure and content remain the same — only the filename or loading mechanism changes.

## How You Use It Day-to-Day

### Before flying a mission

1. Open a terminal in the project directory
2. Launch Claude Code: `claude`
3. Type: `briefing`

Claude silently runs `post_mission.py fix-lang` first (generates the PWCG language files for the mission), then reads the PWCG mission data and squadron context, and delivers the briefing in character as the IO. You can ask questions — the IO answers in character. When you're ready: "Going to fly." The IO closes: *"Understood. See you at the debrief."*

### After landing

1. Back in Claude Code, type: `debrief`

Claude runs the log parsers silently, then the IO starts questioning you — one phase at a time:
- What were your initial conditions?
- Describe the engagement sequence.
- Account for your claims.
- Any casualties in the flight?

As you talk, Claude is cross-referencing your account against the actual logs. Small discrepancies are fog of war. Large ones get flagged. At the end, confirmed kills are announced, losses recorded.

The combat report is filed automatically. Pilot sheets are updated. The journal gets an entry. Finally, `post_mission.py journal` injects a summary into PWCG's own `CampaignLog.json`, so the in-game campaign log stays in sync.

### Between missions

Type: `dispersal`

Claude selects 2-4 pilots who would plausibly be in the dispersal hut given the context, sets the scene briefly, and lets the scene run. Pilots have personalities that emerged from their histories — a veteran who lost a wingman today is not the same as a veteran on a quiet evening. You can talk, or just watch.

## What Is Working and What Is Not

### Working

- Briefing generation from PWCG MissionData files
- Interactive debriefing with log cross-reference
- Combat report generation (RAF Form style)
- Dispersal hut scenes with emergent pilot personalities
- Persistent pilot dossiers updated after each mission
- Squadron journal in a consistent literary register
- Squadron status dashboard

### Not Yet Validated / Needs Work

- The Python parsers have a known limitation: `parse_mission_report.py` can return zero kills even when several were scored, when enemy aircraft are defined as groups (AType:11). The workaround is reading the raw logs directly — which the AI does by default — but someone adapting this project should be aware that the script output is not always reliable as a standalone source
- Enemy pilot dossiers (for killed opponents) are planned but not implemented
- The dispersal pilot rotation system (arrivals, transfers, departures from PWCG) needs refinement
- There is no error recovery if the AI produces malformed output — you would need to manually correct files
- The whole thing only works through Claude Code's CLI — there is no GUI, no web interface, no app

### The Honest Assessment

This module works well when everything is in order: you have the logs, the PWCG files are in the right place, and you run the workflows in sequence. When something is missing or unexpected, it degrades gracefully (contextual output without data) but can also produce gaps you have to fill manually.

It is also entirely dependent on the AI producing coherent, consistent output — which it does reliably in my experience, but which is never guaranteed. An LLM can occasionally produce an off-register line, invent a detail, or misread a data file. The human in the loop (you) is always the final editor.

The project works best if you think of Claude not as an automated system but as a very well-briefed collaborator who reads your data and writes for you. The "automation" is really very structured prompting.

## For the Technically Curious — How the CLAUDE.md Works

The file `CLAUDE.md` at the root of the project is the most important file. It is not documentation for humans — it is documentation for Claude. Every time you start a Claude Code session in this directory, Claude reads this file first.

It contains:
- What the project is and what data sources exist
- Where every file is and what it contains
- The exact sequence of steps for each workflow (briefing, debrief, dispersal)
- The rules: never invent data, never name enemies before combat, always use feet and mph, maintain the register
- The narrative tone guidelines for each type of document

This is called a **system prompt** or **project instructions** in AI terminology. It is the difference between an AI that produces generic military fan-fiction and an AI that behaves like a consistent character in a coherent world. The quality of the output is directly proportional to the quality of these instructions.

### The llm-wiki concept — credit where it's due

The idea of using a set of plain Markdown files as the primary interface between a project and an AI is directly inspired by [Andrej Karpathy's llm-wiki concept](https://github.com/karpathy/llm-wiki). Karpathy's observation, roughly stated, is that well-structured plain text is the most LLM-friendly knowledge format that exists — it requires no special tooling, survives any model change, and can be read and edited by a human just as easily as by an AI.

In this project, `CLAUDE.md` is the wiki entry for the entire module: it tells the AI what the world looks like, what the rules are, and what to do. The files in `resources/` are specialist entries — one per workflow. The Markdown files in `missions/`, `personnel/`, and `squadron/` are the living record that the AI reads at the start of each session to reconstruct the state of the campaign.

No database, no embeddings, no vector store. Just text files on a hard drive, which is exactly the point.

If you fork this project and adapt it for a different campaign, different squadron, or different war — almost everything you need to change is in `CLAUDE.md` and the `resources/` prompts.

## Repository

The project repository contains all the files described above, including the current state of the campaign (two missions, four pilots, one wounded, one status unknown).

The `scripts/` directory contains the Python parsers. The `resources/` directory contains all the system prompts. Everything else is campaign output.

To use it for your own campaign, you would:
1. Clone or download the repository
2. Create a `.env` file with your IL-2 and PWCG paths
3. Clear the `missions/`, `personnel/`, and `squadron/` directories (or keep them for reference)
4. Update `CLAUDE.md` with your pilot's name and squadron
5. Open Claude Code and start
